Raw VulnerabilityReports z każdego klastra. Schemat: CVE ID, severity, image, namespace, cluster, timestamp.
partitions: per cluster · retention: 30d

Raw Falco events ze wszystkich klastrów. Schemat: rule, priority, pod, namespace, cluster, cloud, timestamp.

K8s audit logs (kto co zrobił w API Server). Źródło dla anomaly detection i compliance.
partitions: per cluster · retention: 90d

CVE Enrichment Job
Konsumuje sec.cve.raw. Deduplikuje te same CVE z wielu klastrów (ten sam image, ten sam CVE ID). Normalizuje severity do wspólnego schematu. Dodaje metadane klastra z rejestru.
→ Bronze layer (data lake)
Runtime Alert Router
Konsumuje sec.runtime.raw. Filtruje szum (allowlist dla znanych operatorów). CRITICAL/HIGH → topic sec.alerts.critical → SIEM. Wszystko → data lake Bronze.
→ sec.alerts.critical + Bronze layer
CVE Aggregation Job (micro-batch)
Co 1h: agreguje CVE counts per klaster / severity / namespace / image. Wynik trafia do Silver layer jako Parquet. Podstawa dla PowerBI direct query i CSV exportu.
→ Silver layer (Parquet, partitioned)
Compliance Score Job
Liczy % workloadów bez CRITICAL CVE per klaster. Trend 7d/30d. Wejście dla dashboardu PowerBI "SLA compliance". Aktualizacja co 4h.
→ Gold layer (aggregated KPIs)

Warstwa serwowania — query engine między data lake a outputami

Azure Synapse / AWS Athena / Trino

SQL na Parquet w data lake bez ładowania danych. PowerBI łączy się przez DirectQuery. Zapytania ad-hoc dla security team. Brak konieczności ETL do osobnego DW.

Apache Spark (batch jobs)

Cykliczny job (cron) generuje CSV/Excel z Silver/Gold layer. Agregacja wszystkich CVE z całego środowiska do jednego pliku. Wysyłka na e-mail lub do S3/SharePoint.

SIEM Kafka Connector

Konsument topicu sec.alerts.critical. Splunk Kafka Connect, Sentinel Event Hub connector lub QRadar DSM. Tylko alerty po filtracji Flink — zero szumu.

Warstwa wyjściowa — raporty, SIEM, dashboardy

PowerBI — Dashboardy

  • DirectQuery na Gold layer przez Synapse/Databricks
  • CVE count per klaster / severity / namespace
  • Trend: nowe vs zamknięte CVE (7d/30d/90d)
  • Compliance score: % klastrów bez CRITICAL
  • Heatmapa: które namespace'y są najbardziej podatne
  • Runtime alert frequency per klaster i regułę Falco
  • Row-level security: team widzi tylko swoje klastry

SIEM — Runtime Events

  • Splunk — Kafka Connect plugin (HEC lub direct)
  • Sentinel — Azure Event Hub (Kafka-compatible API)
  • QRadar — Kafka DSM lub syslog bridge
  • Elastic — Kafka Logstash input
  • Tylko eventy po filtracji Flink (zero false positives od operatorów)
  • Enriched: cluster, namespace, pod, cloud, team owner

CVE Report CSV / Excel

  • Spark job (cron tygodniowy/miesięczny)
  • Query na Silver layer: jeden plik, wszystkie klastry
  • Kolumny: CVE ID, severity, image, klaster, namespace, team, fixed_version, days_open
  • Deduplikacja: ten sam CVE w 50 klastrach = 1 wiersz z listą klastrów
  • Export → S3/ADLS → SharePoint → e-mail do security team
Dlaczego Kafka zamiast DefectDojo — kluczowe różnice
Replayability: Kafka przechowuje surowe eventy przez 7–30 dni. Jeśli Flink job miał buga i błędnie klasyfikował eventy — możesz go naprawić i przeprocesować historię od początku. DefectDojo tego nie daje — raz zaimportowany wynik jest finalny.
Skalowalność: Kafka obsługuje miliony eventów na sekundę. Przy setkach klastrów i częstych skanach Trivy (np. po każdym deploy) DefectDojo API staje się bottleneckiem. Kafka buforuje surge'y i wygładza load na downstream.
Data lake vs aplikacja: DefectDojo to aplikacja z własnym modelem danych. Data lake to surowe dane w Parquet — możesz odpytywać dowolnym SQL, budować dowolne agregacje, łączyć z innymi źródłami (billing, CMDB, Jira). PowerBI direktnie na Parquet bez żadnego pośrednika.
SIEM bez szumu: W architekturze DefectDojo każdy klaster wysyła alerty bezpośrednio do SIEM przez Falcosidekick — tuning reguł per klaster jest trudny. Tu Flink robi centralny tuning raz i eliminuje false positives zanim trafią do SIEM. Allowlist dla Kubescape, Trivy, systemowych podów — w jednym miejscu.
Koszt i złożoność: Ta architektura jest znacząco bardziej złożona niż DefectDojo + Falcosidekick. Kafka wymaga operacyjnego doświadczenia. Flink jobs to kod który trzeba pisać i utrzymywać. Data lake wymaga governance (schema, partitioning, retention). Sensowna od ~50+ klastrów i zespołu który umie data engineering. Poniżej tej skali — DefectDojo + Falcosidekick wygrywa prostotą.
